# invest_bot agent guide

## 1. 프로젝트 목적

이 프로젝트의 목표는 투자 전략을 선택하고 자동으로 매매를 수행하는 Python 기반 투자 봇을 개발하는 것이다.

에이전트는 아래 목표를 기준으로 작업한다.

- 사람이 이해할 수 있는 전략 정의 방식 마련
- 전략 검증을 위한 백테스트 흐름 확보
- 실거래 전 모의투자 및 안전장치 우선 적용
- 자동 매매에 필요한 주문, 잔고, 시세 조회 기능 구성
- 유지보수 가능한 구조로 프로젝트를 확장

## 2. 개발 원칙

- 주 개발 언어는 Python으로 한다.
- 구현 전 `reference/` 디렉터리의 자산을 먼저 조사하고 재사용 가능한 코드를 우선 검토한다.
- 단일 스크립트 중심이 아니라 역할이 분리된 모듈 구조를 지향한다.
- 실거래 기능보다 전략 검증, 로그, 예외 처리, 안전장치를 먼저 마련한다.
- 민감 정보는 코드에 하드코딩하지 않고 환경 변수 또는 별도 설정 파일로 관리한다.
- 에이전트는 임의 추측으로 API를 구현하지 말고, reference 문서와 예제를 근거로 작성한다.

## 3. 참고 자산 사용 원칙

우선적으로 참고할 경로는 아래와 같다.

- `reference/open-trading-api/README.md`
- `reference/open-trading-api/docs/`
- `reference/open-trading-api/examples_llm/`
- `reference/open-trading-api/examples_user/`
- `reference/open-trading-api/backtester/`
- `reference/open-trading-api/strategy_builder/`

참고 자산 활용 규칙:

- API 인증, 주문, 시세 조회는 예제 코드를 먼저 확인하고 프로젝트 구조에 맞게 재구성한다.
- 예제 코드를 그대로 복붙하지 말고, 프로젝트 전용 모듈로 분리하며 의존 관계를 정리한다.
- reference와 현재 프로젝트 코드가 충돌할 경우 현재 프로젝트 구조와 테스트 가능성을 우선한다.
- 불분명한 동작은 예제 코드, 문서, 설정 파일 순서로 교차 확인한다.

## 4. 권장 프로젝트 구조

초기 구조는 아래 형태를 권장한다.

```text
invest_bot/
  agent.md
  README.md
  reference/
  src/
    invest_bot/
      __init__.py
      config/
      clients/
      market/
      strategy/
      backtest/
      trading/
      risk/
      jobs/
      utils/
  tests/
  scripts/
  logs/
```

역할 정의:

- `config/`: 환경 변수, 설정 로더, 계좌 및 실행 환경 구분
- `clients/`: 브로커 API 연동, 인증, 요청 래퍼
- `market/`: 종목 정보, 시세 조회, 데이터 정규화
- `strategy/`: 전략 인터페이스, 신호 생성, 전략 조합
- `backtest/`: 전략 검증, 성과 계산, 리포트 생성
- `trading/`: 주문 실행, 체결 확인, 포지션 관리
- `risk/`: 손절, 익절, 주문 한도, 거래 중단 규칙
- `jobs/`: 주기 실행 엔트리포인트
- `utils/`: 공통 유틸리티, 로깅, 시간 처리

## 5. 개발 단계

에이전트는 아래 순서로 작업하는 것을 기본 원칙으로 한다.

### Phase 1. 요구사항 구체화

- 대상 시장 정의: 국내주식, 해외주식, ETF 등
- 전략 유형 정의: 모멘텀, 평균회귀, 돌파 등
- 거래 주기 정의: 일봉, 분봉, 실시간
- 실거래 여부 정의: 백테스트만, 모의투자, 실거래

### Phase 2. reference 분석

- 필요한 API 목록 정리
- 인증 방식, 요청 파라미터, 응답 구조 요약
- 재사용 가능한 예제 파일 경로 기록
- 프로젝트에 가져올 최소 기능 범위 확정

### Phase 3. 최소 기능 제품 구현

최초 MVP 범위는 아래를 권장한다.

- 인증 처리
- 시세 조회
- 계좌/잔고 조회
- 단일 전략 신호 생성
- 모의 주문 또는 주문 시뮬레이션
- 실행 로그 저장

### Phase 4. 백테스트 구축

- 가격 데이터 수집 또는 로딩 방식 정의
- 전략 입력과 출력 형식 통일
- 성과 지표 수익률, MDD, 승률, 샤프 비율 계산
- 전략 비교가 가능한 리포트 형식 확정

### Phase 5. 자동매매 안정화

- 주문 전 체크리스트 적용
- 중복 주문 방지
- 장 시간 외 동작 제한
- 예외 상황 재시도 및 알림
- 손실 제한 규칙 적용

## 6. 에이전트 작업 규칙

에이전트는 다음 규칙을 따른다.

1. 구현 전에 관련 reference 파일과 현재 코드를 먼저 확인한다.
2. 작업 시작 시 목표, 범위, 수정 대상 파일을 짧게 정리한다.
3. 큰 기능은 아래 단위로 쪼개서 진행한다.
   - API 연동
   - 데이터 모델링
   - 전략 로직
   - 백테스트
   - 주문 실행
   - 리스크 관리
4. 각 작업 완료 후 아래 중 최소 하나를 남긴다.
   - 테스트 코드
   - 실행 스크립트
   - 사용 예시
   - 로그 예시
5. 실거래 관련 코드는 반드시 안전장치를 포함한다.
6. 설정값과 비밀키는 `.env` 또는 별도 설정 파일로 분리한다.
7. 기능 추가 시 README 또는 관련 문서를 함께 갱신한다.

## 7. 의사결정 기준

여러 선택지가 있을 때는 아래 우선순위를 따른다.

1. 안전성
2. 검증 가능성
3. 단순성
4. 확장성
5. 실행 성능

특히 자동 매매에서는 "더 많은 기능"보다 "실수 방지"를 우선한다.

## 8. 금지 사항

- reference 예제를 검증 없이 그대로 실거래에 연결하지 않는다.
- 비밀키, 계좌번호, 토큰을 코드나 저장소에 커밋하지 않는다.
- 테스트 없이 주문 로직을 확장하지 않는다.
- 전략과 주문 로직을 강하게 결합하지 않는다.
- 예외를 무시한 채 재시도 루프를 만들지 않는다.

## 9. 우선 구현 후보

프로젝트 초기에는 아래 순서가 적절하다.

1. 프로젝트 기본 구조 생성
2. 설정 로더와 로깅 구성
3. reference 기반 인증 클라이언트 구현
4. 시세 조회 모듈 구현
5. 단순 전략 인터페이스 정의
6. 백테스트 실행기 초안 구현
7. 모의투자 주문 흐름 연결
8. 리스크 룰 및 알림 추가

## 10. 산출물 기대치

에이전트는 기능 구현 시 아래 산출물을 함께 만드는 것을 목표로 한다.

- 실행 가능한 Python 코드
- 최소 1개 이상의 검증 방법
- 필요한 설정 예시
- reference 출처 메모
- 다음 단계 제안

## 11. 초기 작업 지시 예시

아래와 같은 방식으로 에이전트에게 작업을 요청한다.

- `reference/open-trading-api/examples_user`를 기준으로 인증 및 국내주식 시세 조회 모듈을 작성해줘.
- 전략 인터페이스를 설계하고 이동평균 크로스 전략 예제를 추가해줘.
- 백테스트 모듈의 기본 구조와 테스트 가능한 샘플 데이터를 만들어줘.
- 주문 실행 전 위험 제한 규칙을 넣고, 모의투자 모드만 동작하도록 구성해줘.

## 12. 현재 프로젝트에 대한 기본 해석

현재 저장소는 루트 코드가 거의 없고 `reference/open-trading-api`가 핵심 참고 자산이다.

따라서 초기 개발 방향은 아래와 같이 해석한다.

- reference 분석 결과를 바탕으로 프로젝트 전용 래퍼를 만든다.
- 바로 자동매매 전체를 구현하지 말고 백테스트와 모의 환경부터 만든다.
- 전략 엔진과 주문 엔진을 분리해 추후 전략 교체가 쉽도록 만든다.
- 에이전트는 "빠른 구현"보다 "검증 가능한 점진적 개발"을 우선한다.
